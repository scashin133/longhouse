Note

In this project, user roles are hierarchical, as explained in the glossary. In these use cases, unless specifically noted otherwise, an action available to a user of a specific type is also available to all other users above that user in the hierarchy.

Default Aspects of All Use Cases

Direct Actors:
User: end-user in any role
System: The system being built
When actors are not listed, assume User is doing it.
Items beginning with "see" indicate that System has presented a new screen.
Stakeholders: Project Owners and other members
Prereq: User is logged in

UC-101: Log in to account

Summary: Anonymous visitor with a Google account logs in using Google AuthSub.
Use frequency: Often
Direct actors: Anonymous Visitor
Preconditions: Anonymous visitor is not logged in
Main success scenario:
  1. Anonymous visitor clicks link to log in
  2. Anonymous visitor is sent to Google site where they log in
  3. Anonymous visitor is sent back to our system
Success postconditions: This user's credentials are now known to our system. They are now recognized as a System Administrator, Project Owner, Project Member, or Identified Visitor.
Exceptions:
  • Anonymous visitor does not have a Google account
  • Anonymous visitor does not enter corrent login information
  • Anonymous visitor does not grant our system access to their Google account
Failure postconditions: Anonymous visitor remains not logged in

UC-103: Log out of account

Summary: User logs out of system.
Use frequency: Often
Preconditions: User is logged in
Main success scenario:
  1. User clicks link to log out
  2. User is brought to page notifying them that they have been successfully logged out
Success postconditions: User is now identified as an anonymous visitor

UC-200: Create a Project

Summary: A logged-in user creates a new project.
Use frequency: Sometimes
Direct actors: Identified visitor
Preconditions: Identified visitor is logged in
Main success scenario:
  1. User clicks to create a new project
  2. User enters the name for the new project, a project summary, description of the project, choosing a license, and adding labels to the project.
  3. User submits request to create project.
  4. System creates project and takes creator to project home page.
Alternative success scenario:
  1. The project creator chooses a project name that is already used. When the project creation form is submited, the project creator is brought back to the project creation form and asked to chose another project name. Valid fields from the previous form are automatically filled in for the new form.
  2. The project creator does not enter one or more fields of information. When the project creation form is submited, the project creator is brought back to the project creation form and asked to enter information for those fields that were missing. Valid fields from the previous form are automatically filled in for the new form.
Success postconditions: The project is successfully created and allocated disk space, the project creator is registered as the initial project owner, and is taken to the project's home page.
Exceptions:
  • The project creator has already created 10 projects. Project creation is denied.
  • The disk space of the server is full. Project creation is denied.
Failure postconditions: The project creator's request to create the project is denied when he or she attempts to submit the project's information for creation.

UC-201: Search for a Project

Summary: A user uses the system search engine to find a project.
Use frequency: Often
Direct actors: User: a user in any role
Preconditions: None
Main success scenario:
  1. User enters the name or keyword of a project to find.
  2. The system displays the projects that match the name or keyword the user has entered
  3. The user can click any of the search results to visit that project's home page.
Alternative success scenario:
  1. The system presents the user with a list of labels. Clicking a label allows the user to see the projects that are associated with that label, and clicking a project's name will take the user to that project's home page.
Success postconditions:
  • The user finds the particular project they were searching for.
Exceptions:
  • The user's search doesn't return any results.
  • The user's search doesn't return the result(s) they were expecting.
Failure postconditions: The user does not find the project they were searching for.

UC-202: Join a Project

Summary: An identified visitor chooses to join a project.
Use frequency: Sometimes
Direct actors: Identified visitor
Preconditions: Identified visitor has found the project home page.
Main success scenario:
  1. Identified visitor clicks to join the project currently being viewed.
  2. The system presents instructions to the user to participate in the project's mailing list and request being added to the project via the mailing list.
Success postconditions: Identified visitor is aware of the procedure for joining a project.

UC-300: Change project information

Summary: A Project Owner changes the information for a project he or she owns.
Use frequency: Sometimes
Direct actors: Project Owner
Preconditions: Project Owner is logged in and on the project's home page.
Main success scenario:
  1. User enters the project administration section of the website.
  2. User edits the summary of the project, description of the project, license the project uses, labels attached to the project, URLs for blogs and miscellaneous websites, discussion group email addresses, email addresses to notify when activity occurs in the project, and the means of tracking project statistics.
  3. User submits the request to save the changes.
Alternative success scenario:
    Case A:
  1. User logs into the Subversion repository using his or her username and generated password.
  2. User downloads the settings file for the project and edits the file.
  3. User reuploads the changed file to the repository.
Success postconditions: Settings are successfully saved and used by the project.

UC-301: Add new members to project

Summary: A Project owner adds a new member to the project.
Use frequency: Somewhat frequently
Direct actors: Project owner
Preconditions: Project is created and multiple users exist in the system.
Main success scenario:
  1. User visits the home page of the project for which he/she is an owner.
  2. User enters Administration area of project site.
  3. User enters Project Members tool in Administration section.
  4. User enters the user names of users to add to the project membership roster.
  5. User chooses to save data and receives an alert that data was saved.
Success postconditions: Additional members are added to the project roster and can create issues, Wiki pages and downloads.

UC-302: Promote members of project

Summary: A Project owner promotes project members to project owners.
Use frequency: Somewhat frequently
Direct actors: Project owner
Preconditions: Project is created and multiple users exist in the system, and UC-301 has happened at least once.
Main success scenario:
  1. User visits the home page of the project for which he/she is an owner.
  2. User enters Administration area of project site.
  3. User enters Project Members tool in Administration section.
  4. User enters the user names of users who are already project members to add to the project ownership roster.
  5. User chooses to save data and receives an alert that data was saved.
Success postconditions: Additional owners are added to the project roster and can create issues, Wiki pages and downloads. In addition they can also execute UC-301 and UC-302.

UC-303: Delete project

Summary: A Project owner flags project for deletion.
Use frequency: Infrequent
Direct actors: Project owner
Preconditions: Project is created and the user owns the project.
Main success scenario:
  1. User visits the home page of the project for which he/she is an owner.
  2. User enters Administration area of project site.
  3. User enters Advanced Options tool in Administration section.
  4. User chooses to flag project for deletion.
  5. System alerts user about the consequences of deletion.
  6. User clicks to affirm the choice to flag for deletion.
  7. System shows notice that the project is hidden and flagged for deletion.
Success postconditions: System shows alert that the project is flagged for deletion.

UC-400: Search for a file

Summary: A User searches for a file to view.
Use frequency: Often
Direct actors: User
Preconditions: None
Main success scenario:
  1. User enters the files portion of the project home page.
  2. User is presented with a list of files with details in columns.
  3. User optionally clicks the column names to sort the list by that column.
  4. User clicks the summary of the file to view information for that file.
  5. The system shows the user a download link, who uploaded the file, when it was uploaded, a checksum, any labels attached to it, and how many times it was downloaded. From this page the user can perform UC-401.
Alternative success scenario:
    Case A:
  1. User enters the files portion of the project home page.
  2. User types a search term into the search box.
  3. User is presented with a list of files on that project that match the query.
  4. User can click the summary of one of the files in the list to view that file, enter a new search, or go back to the complete list of files to search another way.
    Case B:
  1. Instead of clicking on the summary on the file list page, the user clicks the file name, which performs UC-401.
    Case C:
  1. User uses a Subversion client to find a file (see UC-700).
Success postconditions: User has found a file to view and can execute UC-401.

UC-401: Download a file

Summary: A user can download a specific file.
Use frequency: Often
Direct actors: User
Preconditions: A user has performed UC-400
Main success scenario:
  1. User clicks the download link to download the file
  2. The system serves the file to the user.
Alternative success scenario:
    Case A:
  1. The user uses a Subversion client to download the file (see UC-700).
Success postconditions: User successfully receives the file.

UC-402: Upload a file

Summary: A user uploads a new file that other users can download.
Use frequency: Sometimes
Direct actors: Project Member
Preconditions: None
Main success scenario:
  1. Project Member selects to upload a new file.
  2. Project Member supplies the file, enters a one-line summary of the file, and adds labels to the file.
  3. Project Member submits the file and it is successfully uploaded, and appears on the file list page.
Alternative success scenario:
    Case A:
  1. The file the Project Member submits is larger than 20 MB. The system requests the user to choose a different file.
  2. The Project Member chooses a different file to upload and uploads successfully.
    Case B:
  1. The Project Member logs into the repository using a Subversion client and uploads the file in that client.
Success postconditions: The file is succesfully uploaded and visible in the repository and on the file list page.
Exceptions:
  • The user needs to upload a file larger than 20MB. The action is not allowed by the system and the file cannot be hosted.
Failure postconditions: The user's desired file is not uploaded.

UC-403: Change information related to a download

Summary: A Project Member changes the information related to a download.
Use frequency: Rarely
Direct actors: Project Member
Preconditions: Project Member has performed UC-400 and is viewing the download's information page.
Main success scenario:
  1. Project Member supplies a new summary and a new set of labels.
  2. Project Member submits changes.
Alternative success scenario:
  1. Project Member logs onto the repository using a Subversion client and downloads the file's metadata.
  2. Project Member edits the metadata file for that download and changes the summary and labels.
  3. Project Member re-uploads the metadata file.
Success postconditions: The file's information is successfully changed.

UC-404: Change download settings

Summary: A project owner changes the settings of the downloads section of the project home page.
Use frequency: Rarely
Direct actors: Project Owner
Preconditions: None
Main success scenario:
  1. Project Owner enters the administration section of the project home page.
  2. Project Owner chooses to edit downloads settings.
  3. Project Owner can delete or edit the existing labels or add new labels, change the prefixes that a download can have only one of, and change the order that file details are shown in on the file list page.
  4. Project Owner submits changes.
Alternative success scenario:
  1. Project Owner logs into Subversion, downloads the config file for the downloads section, makes changes, and uploads the new version of the config file via Subversion.
Success postconditions: The new settings are applied to the downloads section.

UC-500: Create a new wiki page

Summary: Project member creates a new wiki page.
Use frequency: Sometimes
Direct actors: Project member
Main success scenario:
  1. User goes to the wiki page list
  2. User clicks the new page link
  3. User is brought to wiki edit page
  4. User enters a new page name, content, and optional labels (see UC-502)
  5. User clicks save page button
Alternative success scenario:

Precondition: An existing wiki page contains a link to another wiki page that does not exist

    Case A:
  1. User clicks the link to the nonexistant wiki page
  2. User is brought to wiki edit page, with the page name already populated
  3. User completes and saves the page the same as in the main success scenario
    Case B:
  1. User creates the file for a new Wiki page and uploads it via Subversion.
Success postconditions: The new wiki page is created, and can now be seen in the wiki page list.
Exceptions:
  • User creates a new page with the same name as an existing page (can only happen in the main success scenario)
Failure postconditions: New wiki page is not created. User is brought to a page notifying them of why their page was not created.

UC-501: Search for a wiki page

Summary: User searches for a wiki page, either by browsing the list of pages or using the search box.
Use frequency: Often
Preconditions: User is at the wiki page list
Main success scenario:
  1. User looks through the wiki page list and sees the page she is looking for
  2. User clicks the link to that wiki page
Alternative success scenario:
  1. User enters a search string into the search box
  2. User clicks the search button
  3. User is presented with a list of wiki pages that match that search criteria. One of these is the desired page.
  4. User clicks the link to that wiki page
Success postconditions: User is brought to the desired wiki page
Exceptions:
  • User does not locate the page they were looking for
Failure postconditions: User is sad

UC-502: Edit a wiki page

Summary: Project member edits a wiki page.
Use frequency: Often
Direct actors: Project member
Preconditions: User has found and is viewing the wiki page they wish to edit (see UC-501)
Main success scenario:
  1. User clicks the link to edit the page
  2. User is brought to the "edit wiki page" page
  3. User modifies content and/or labels
  4. User specifies a commit comment (a default one is provided)
  5. User clicks button the save the page
Alternative success scenario:
  1. User uses a subversion client to check out a copy of the page from the repository
  2. User edits the contents of this file
  3. User commits the changes back to the repository
Success postconditions: The wiki page has been changed
Exceptions:
  • While user is editing the page, another user edits and saves the same page.
Failure postconditions: User is brought to a page notifying them a collision has occured.

UC-503: Delete a wiki page

Summary: A project member deletes a wiki page.
Use frequency: Rarely
Direct actors: Project member
Preconditions: User has found and is viewing the wiki page they wish to delete (see UC-501)
Main success scenario:
  1. User clicks the link to delete the page
Success postconditions:
  • The user is brought to a page notifying them the page was deleted.
  • The page no longer shows up in the web interface but is still in the repository.

UC-504: Comment on a wiki page

Summary: A project member adds a comment about a wiki page.
Use frequency: Often
Direct actors: Project member
Preconditions: User has found and is viewing the wiki page they wish to comment on (see UC-501)
Main success scenario:
  1. User enters their comment text in the comment box
  2. User clicks the button to submit their comment
Alternative success scenario:
  1. User enters the Subversion repository, edits the Wiki page's comments and reuploads the new version of the file via Subversion.
Success postconditions: A new comment can be seen when viewing that wiki page

UC-505: Change wiki settings

Summary: A project owner changes the wiki settings for a given project.
Use frequency: Rarely
Direct actors: Project owner
Main success scenario:
  1. User goes to the project administration section
  2. User goes to the wiki administration area
  3. User makes changes to one or more of the settings displayed
  4. User clicks a button to apply the changes
Alternative success scenario:
  1. User enters the Subversion repository, edits the config file for the Wiki section of the site, and uploads the new version of the config file via Subversion.
Success postconditions: The project's wiki reflects the new settings

UC-600: Add a new issue

Summary: A Project Member adds a new issue to the list of issues.
Use frequency: Sometimes
Direct actors: Project Member
Preconditions: None
Main success scenario:
  1. Project Member enters the issue section of the website.
  2. Project Member chooses to add a new issue.
  3. Project Member supplies a summary, describes the issue, and sets the status of the issue and the labels attached to the issue. Project Member can optionally specify a different owner than him or herself for the issue, and optionally attach a file.
  4. Project Member submits issue request.
Alternative success scenario:
  1. Project Member creates a new issue file and adds it to the repository using a Subversion client.
Success postconditions: Issue is successfully added to the list of issues.

UC-601: Search for an issue

Summary: A user searches for an issue and is able to view its details.
Use frequency: Often
Direct actors: Anonymous Visitor
Preconditions: At least one issue is created for the project
Main success scenario:
  1. User enters the issue section of the project web site.
  2. User is presented with a list of current issues. He or she may resort the list according to a column or his or her choosing, or enter a search query that returns a list of issues that match the query.
  3. User can click on the issue's summary to view that issue's page, which lists the description, status, owner and labels of the issue, and the comments that users have made on the issue.
Alternative success scenario:
    Case A:
  1. Rather than finding a specific issue, the Anonymous Visitor wants to see a visualization of the issues. From the issue list page, the user selects the Grid view.
  2. The user is able to choose an attribute of the issues to use as the row delineation for the Grid view, and another attribute for the column delineation, as well as what each cell in the Grid will show.
  3. The user submits the request to order the issues in the Grid.
  4. The user can then click on an issue's cell to view its description page.
Success postconditions: The user has successfully found the issue he or she was looking for.
Exceptions:
  • The issue the user has been looking for does not exist.
Failure postconditions: The user does not find the issue he or she was looking for.

UC-602: "Star" an issue

Summary: A user chooses an issue to 'star', meaning that when that issue is changed, the user will receive an email reporting the change.
Use frequency: Rarely
Direct actors: Identified Visitor or higher
Preconditions: There is at least one issue in the list of issues.
Main success scenario:
  1. The user enters the issue section of the project website.
  2. While viewing the list of issues, the user clicks the star next to the issue's row, and the star illuminates.
  3. The user will now receive an email if that issue is modified.
Alternative success scenario:
    Case A:
  1. Rather than clicking an issue's star on the list of issues, the user clicks the star on an individual issue's description page. The star illuminates.
  2. The user will now receive an email if that issue is modified.
Success postconditions: The user has successfully marked the issues he or she wishes to be notified about when they change, and the system successfully sends an email when those issues are changed.
Exceptions:
  • There are no issues to "star."
  • There is a technical issue receiving the emails sent by the system.
Failure postconditions: The user does not receive emails when desired issues update.

UC-603: Change an issue

Summary: A user changes attributes of an issue.
Use frequency: Sometimes
Direct actors: Project Member or higher
Preconditions: There is at least 1 issue in the issue list
Main success scenario:
  1. The user enters the issue section of the web site.
  2. The user finds the issue to modify (see UC-601).
  3. The user enters the issue's information page.
  4. The user can change the summary of the issue, change the status of the issue, specify a new owner, and change the labels attached to the issue and attach additional labels.
  5. The user submits the request to change the information.
  6. The user is brought back to the issue's information page, which shows the changes he or she made.
Alternative success scenario: User enters the Subversion repository, downloads the issue file, modifies it, and uploads the new version using a Subversion client.
Success postconditions: The information related to the issue is succesfully changed and displayed.

UC-604: Comment on an issue

Summary: A user submits a comment on an issue.
Use frequency: Sometimes
Direct actors: Anonymous Visitor or higher
Preconditions: None
Main success scenario:
  1. The user enters the issues section of the project web site.
  2. The user finds the issue to comment on (see UC-601).
  3. The user views the issue's information page and chooses to add a comment.
  4. The user enters a comment and submits the comment.
  5. The user is taken back to the issue's information page and his or her comment is displayed.
Alternative success scenario: User enters the Subversion repository, downloads the issue's file, edits it to add a comment, and uploads the new version using a Subversion client.
Success postconditions:
  1. The user's comment is displayed successfully.
  2. If the user was an Anonymous Visitor at the time of submission, the comment is shown as attributed to "Anonymous."
  3. If the user was a Identified Visitor or higher user level, the comment is attributed to their username.

UC-605: Change issue tracking settings

Summary: A project administrator changes the settings of the project web site's issue section.
Use frequency: Rarely
Direct actors: Project Administrator or higher
Preconditions: None
Main success scenario:
  1. The user enters the web site's administration section.
  2. The user chooses to change Issues settings.
  3. The user can add to, edit, or delete from the list of status values and labels available to users who start new issues or edit issues.
  4. The user can also change the instructions shown to users when they create new issues and change the list of label prefixes that users may only choose one of when tagging an issue with labels.
  5. The user can also change the sorting of the issue list page, and the default axes of the Grid view.
  6. The user submits the changes to be saved.
Alternative success scenario: The user enters the repository, downloads the config file for the issue section of the website, edits the file to make changes, and uploads the new version using a Subversion client.
Success postconditions: The changes are successfully saved.

UC-700: Browse subversion repository

Summary: Users can browse the Subversion repository for a project through the web interface or a Subversion client.
Use frequency: Often
Direct actors: Anonymous Visitor or Project Member.
Preconditions: None
Main success scenario:
  1. The Anonymous Visitor is clicks a link to browse the Subversion repository via the web.
  2. The Anonymous Visitor is shown the directory layout for the Subversion repository.
  3. The Anonymous Visitor browses the repository by clicking folders in the web view.
Alternative success scenario:
    Case A:
  1. The Anonymous Visitor types in a search term into a search box in the web interface.
  2. The system searches the project's repository and returns the results that match the user's query.
    Case B:
  1. The Anonymous Visitor sees the instructions given on the web interface to anonymously check out the repository using a Subversion client.
  2. The Anonymous Visitor follows the instructions using a separate client to successfully download the repository.
    Case C:
  1. The Project Member sees the instructions given on the web interface to log in and browse the repository as a registered user using a Subversion client.
  2. The Project Member follows the instructions using a separate client to successfully browse the repository.
Success postconditions: The Anonymous Visitor or Project Member have the ability to browse the repository and download files from it one or more ways.
Exceptions:
  • The Project Member has been banned from the service. The Subversion client rejects him/her when he/she attempts to log in via a separate Subversion client.
  • The project is invisible to non-members. The Anonymous Visitor is required to log-in before he or she can browse and download the repository.
Failure postconditions: The user cannot access the Subversion repository in his or her current user state.

UC-800: Promote user to system administrator

Summary: Promote a user account from a normal user account to a system administrator account.
Use frequency: Infrequently
Direct actors: System administrator.
Preconditions: None
Main success scenario:
  1. Enter system administrator interface.
  2. Choose the user control tool.
  3. Find the account to promote.
  4. Choose to promote the account.
Success postconditions: The selected account is promoted. The system administrator is notified of the successful account status change.

UC-801: Ban a user

Summary: The system administrator bans a user account from performing any actions in Longhouse.
Use frequency: Infrequently
Direct actors: System administrator.
Preconditions: None
Main success scenario:
  1. Enter system administrator interface.
  2. Choose the user control tool.
  3. Find the account to ban.
  4. Choose to ban the account.
Success postconditions: The banned account cannot create projects or perform any actions in projects.

UC-802: Allocate a project's disk space

Summary: The system administrator changes the amount of disk space a project is allocated.
Use frequency: Infrequently
Direct actors: System administrator.
Preconditions: None
Main success scenario:
  1. Enter the system administrator interface.
  2. Choose the project control tool.
  3. Find the project to modify.
  4. Choose to change disk allocation.
  5. Set a new disk storage limit.
Success postconditions: The specified project has a new disk quota. If the amount of space the project is using is below or equal to the new limit, new downloads and other things taking up disk space cannot be created.

UC-803: Check site-wide metrics

Summary: The system administrator views a variety of statistics about the site activity.
Use frequency: Infrequently
Direct actors: System administrator.
Preconditions: None
Main success scenario:
  1. Enter the system administrator interface.
  2. Choose the statistics tool.
  3. Choose to view project statistics, view user statistics or view site-wide statiscs.
  4. Choose categories/sub-categories until the desired statistic is found.
Success postconditions: Desired statistic is made available to the system administrator.

UC-1000: View a user's profile

Summary: A user views another user's profile.
Use frequency: Often
Direct actors: User
Preconditions: The User is viewing a page with another user's username.
Main success scenario:
  1. User clicks on the other user's username.
  2. User is shown a page listing the other user's username, the projects for which the other user is a Project Member, and the projects for which the other user is a Project Owner.
Success postconditions: The lists of projects for which the user being viewed is an owner and a member are shown.
