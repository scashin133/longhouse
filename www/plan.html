Project Information

Project: Longhouse
Project Time-frame: September 24, 2007 - June 13, 2008

Summary of Project

This is the senior design project for the Informatics program at UCI. Our goal is to make an open source project hosting solution based off the Google Code Collaborative Development Environment (CDE). This system will be installed on a project's own servers instead of running on Google's servers. In the short term we will be porting the system from Google proprietary technologies to open source solutions. In the future we will introduce some new ideas based off feedback and observations from the open source community.

We hope that this project is not only a single successful open source project, but also produces a system that benefits the open source community as a whole. After June 2008 we will continue this project and promote it as a fully-fledged open source project by getting the word out and allowing other people to contribute their own design, coding and testing skills.

Summary of Methodology

What general development approach will be used?
Will follow the waterfall development structure.
Will follow a test driven development style to prevent errors normally caused by interpreted languages.
Once port is complete will use a much more rapid development cycle but will still include generating or altering a requirements and design document.
How will the project team be organized?
Professor Ziv, the TA, and our customer are the decision makers.
All members of the group will trade off responsibilities. Everyone is expected to be a project leader, coder, requirements gatherer, designer, and a tester.
What development and collaboration tools will be use?
We plan to use the following tools extensively through out the project:
  • Project website

    • http://code.google.com/p/teamfreedom-projectcode

  • Project mailing lists

    • team-freedom@googlegroups.com

  • Issue tracking system

    • http://code.google.com/p/teamfreedom-projectcode/issues/list

  • Version control system

    • http://code.google.com/p/teamfreedom-projectcode/source

How will this plan be updated?
This project plan will be updated as needed throughout the project. It will be placed under version control and instructions for accessing it will be on the project website.

Risk Management

The main risks of this project are:

  1. The code base not being delivered within the two weeks. We will manage this by spending more time requirements gathering and researching for new ideas. There is nothing to be done more than making sure we do not waste time in this event.
  2. There are no design documents for the code base given. We will allocate more time spent studying the code base and will write up our own documentation. Insist on meetings with the writer of the code to help us better understand it.
  3. Team member is unavailable for a long period of time i.e. injury, emergency. Will inform all stakeholders of the event to make sure there is understanding as to why productivity has taken a turn. Distribute work assigned to the missing team member and have all other members pick it up.
