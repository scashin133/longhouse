Feature Set

Release Information

Project: Longhouse
Related Documents: Software Requirements Specification
Project Glossary

Note

The following feature set describes the features of the current Google Code system as hosted on http://code.google.com. As such, this feature set is the metric by which the port of the current system will be evaluated.

Features in this feature set are designated according to whether they require only integration of prefabricated components (N), implementation of new components (M), and a hybrid requirement of implementing alongside the integration of components (H).

Account Management

  1. FS-101 [N]: All users should be able to log into a previously created account.
    • Account creation and account maintenance is accomplished through a separate Google system and thus is not documented here.
  2. FS-102 [M] : All users should be able to log out of their account.
    • Logging out of an account should not require a user name or password.
    • Logging out of an account should take a user back to the page he or she was viewing prior to logging out, but with the subtraction of any options requiring project member or owner privileges, e.g., changing project administrative information.

User/Project Relationship

  1. FS-200 [M]: All users should be able to create a project.
    • Project creators should supply the project name, a summary of the project, and a short description.
    • Project creators should supply labels, which are single words that describe the languages used or areas explored; any information that helps with classification.
    • Project creators must choose from one of several licenses:
      • Apache License v2.0
      • Artistic License/GPLv2
      • GNU General Public License v2
      • GNU General Public License v3
      • GNU Lesser General Public License
      • MIT License
      • Mozilla Public License 1.1
      • New BSD License
    • Project creators can create a maximum of 10 projects.
  2. FS-201 [H]: All users should be able to search for a project.
    • Users should be able to use a search box to search through project names, summaries, descriptions, etc.
    • Users should also be able to browse by project labels.
  3. FS-202 [M]: All users should be able to join one or more projects.

Project Administration

  1. FS-300 [M]: Project administrators should be able to change the information related to a project.
    • The project summary, description, license and labels should be changeable.
    • It should be possible to add links to relevant websites to the front page of the project home page.
    • It should be possible to register Google Groups related to the project and have their mailing addresses appear on the front page of the project home page.
    • It should be possible to add a link to blogs related to the project to the front page of the project home page.
    • It should be possible to send email notifications of project activity to email addresses, project activity being issue changes and subversion commits.
    • It should be possible to register the project with Google Analytics.
  2. FS-301 [M]: Project administrators should be able to change project membership.
    • New members are added by supplying email addresses to either the "project members" or "project owners" fields of email addresses.
    • Existing members are removed by removing their email address.
  3. FS-302 [M]: Project administrators should be able to change user roles.
    • Project members should be able to become project owners by moving their email address to the "project owners" email addresses section.
    • Project owners should be able to be demoted to project members by moving their email address to the "project members" email addresses section.
  4. FS-303 [M]: Project administrators should be able to delete the project.

Downloads

  1. FS-400 [H]: All users should be able to search for a file.
    • Upon clicking on the "download" tab, users should be presented with a list of files with their filename, summary, date uploaded, size and download count.
    • Users should be able to resort this list according to any column to find a file.
    • Users should also be presented with a search box to search the list of files.
    • Users should also be able to search for files by Subversion command-line options or a Subversion client.
  2. FS-401 [M]: All users should be able to download a file.
    • Files should be downloadable from the file list page.
    • Users should be able to click the file's summary on the file list page, which should take them to a page where the file's information is listed, a download link is presented, and a SHA1 checksum for the file is generated and displayed.
    • Users should also be able to download files by using Subversion command-line options or a Subversion client.
  3. FS-402 [M]: Project members should be able to upload a file.
    • Uploaders should supply the file from their hard drive, a summary of the file, and labels to classify the download. See the section on changing the download settings for a list of default labels.
    • Files may also be committed using Subversion command-line options or a Subversion client.
  4. FS-403 [M]: Project members should be able to change information related to a file.
    • The Summary and Labels should be changeable.
  5. FS-404 [M]: Project administrators should be able to change settings related to downloads.
    • Settings include a list of predefined labels, columns and their order displayed on the file list page, and the default sorting order for the file list page.
    • Labels have a name and a comment; the comment of the label is displayed when the user selects label(s) to tag a new download.
    • Labels for downloads optionally have prefixes. The administrators should be able to specify prefixes that a given download may have at most one of, e.g., a given download might be required to have only one Type label and only one OpSys label.
    • Administrators should be able to add, change and delete labels.
    • The following labels are default for downloads:
      • Featured (listed on project home page)
      • Type-Executable (Executable application)
      • Type-Installer (Download and run to install application)
      • Type-Package (Your OS package manager installs this)
      • Type-Archive (Download, unarchive, then follow directions)
      • Type-Source (Source code archive)
      • Type-Docs (This file contains documentation)
      • OpSys-All (Works with all operating systems)
      • OpSys-Windows (Works with Windows)
      • OpSys-Linux (Works with Linux)
      • OpSys-OSX (Works with Mac OS X)
      • Deprecated (Most users should NOT download this)

Wiki

  1. FS-500 [H]: Project members should be able to create a new Wiki page.
    • Wiki page creators should supply a one-word Page Name, Labels, and the page content.
    • Content can include Wiki Markup, which should be shown and explained on the Page Creation screen.
    • Creators should be able to preview a new page before saving it.
    • New pages should be able to be added by using Subversion command-line options or a Subversion client.
  2. FS-501 [H]: All users should be able to search for a Wiki page and view Wiki pages.
    • Users should be presented with a list of Wiki pages when they click on the "Wiki" tab. The list should list the Page Names, Summary and Labels, Date Last Changed, and who changed it.
    • Users should also be able to use a search box to find a Wiki page.
    • The contents of the Wiki page should be viewable by everyone.
    • Users should also be able to retrieve a Wiki page by using Subversion command-line options or a Subversion client.
  3. FS-502 [H]: Project members should be able to edit a Wiki page.
    • Editors should be able to use Wiki Markup and preview a page, similar to creators of new Wiki pages.
    • Changes to Wiki pages should be able to be committed by using Subversion command-line options or a Subversion client.
  4. FS-503 [H]: Project members and/or visitors should be able to comment on a Wiki page.
    • The interface to comment should be at the bottom of each Wiki page.
    • Comments should be listed in order of date entered, from earliest to latest, with the commentor's name.
    • Comments should be able to use Wiki Markup.
    • Visitors may be able to add comments to Wiki pages. This depends on whether an administrator has enabled or disabled the option.
    • Project administrators should be able to delete a comment.
  5. FS-504 [H]: Project administrators should be able to delete a Wiki page.
    • The Wiki page should still exist in the Subversion repository but not be viewable by the browser interface.
  6. FS-505 [H]: Project administrators should be able to change settings related to Wiki pages.
    • Labels for Wiki pages have a name and a comment; the comment is shown to users to describe the label's meaning when they are tagging Wiki pages.
    • Label names optionally have prefixes; administrators should be able to specify that a Wiki page can only have one tag with a specified prefix. For example, an administrator might specify that a page have only one tag that begins with "Phase."
    • The following labels for Wiki pages are default:
      • Featured (Listed on project home page)
      • Phase-Requirements (Project vision and requirements)
      • Phase-Design (Project design and key concerns)
      • Phase-Implementation (Developers' guide)
      • Phase-QA (Testing plans and QA strategies)
      • Phase-Deploy (How to install and configure the program
      • Phase-Support (Plans for user support and advocacy)
      • Deprecated (Most users should NOT reference this)
    • Project administrators should be able to choose the columns shown on the Wiki page list and their order, as well as their sorting order.
    • Project administrators should be able to turn on and off the ability for visitors to add comments to Wiki pages.

Issues

  1. FS-600 [M]: Project members should be able to add a new issue.
    • A new issue requires a summary and a short description.
    • Issue adders should be able to specify who owns the issue; it does not necessarily belong to them although that is the default state.
    • Issue adders should be able to supply labels for the issue.
    • Issues require the issue adder to choose a status for the issue. Please see the section on changing settings related to issues for a list of status labels.
    • Users should be able to add an issue using Subversion command-line options or a Subversion client.
  2. FS-601 [H]: Users should be able to search for an issue and view its details.
    • When they click on the "Issues" tab, users should be presented with a list of issues: their ID number, their type, status, priority, milestone date, owner, and summary and labels.
    • Clicking on the row that an issue inhabits should take a user to the page for that issue.
    • On the issue's page, it should list ID number, status, owner, labels, the initial description, and any comments.
    • Clicking on a label of that issue should bring up a list of issues that have a matching label.
    • Users should also be able to search for an issue via a search box.
    • Users should also be able to search for an issue by using Subversion command-line options or a Subversion client.
  3. FS-602 [M]: Project members should be able to comment on an issue and change its information.
    • The interface to add comments should be on the issue's dedicated page.
    • Existing comments should be listed by date, most recent to least recent.
    • When adding a comment, project members should also be able to change the issue's summary, status, owner and labels.
    • Project administrators should be able to delete comments.
    • Commentators should be able to attach a file to their comments for other users to download.
  4. FS-603 [M]: Project members should be able to "Star" an issue.
    • When a user "stars" an issue, they are sent an email by the system when that issue is changed.
  5. FS-604 [M]: Project administrators should be able to change settings related to issues.
    • Project administrators should be able to define status values and their meanings. This includes adding new status values and deleting from the existing pool of status values, as well as changing the meaning of existing values. The following values are default:
      • New (Issue has not had initial review yet)
      • Accepted (Problem reproduced/Need acknowledged)
      • Started (Work on this issue has begun)
      • Fixed (Developer made requested changes, QA should verify)
      • Done (Task has been completed)
      • Verified (QA has verified that the fix worked)
      • Invalid (This was not a valid issue report)
      • Duplicate (This was not a valid issue report)
      • WontFix (We decided not to take action on this issue)
    • Project administrators should be able to define labels, and set a uniqueness clause for labels with given prefixes.
    • Project administrators should be able to write instructions that the system will supply to users creating a new comment.
    • Project administrators should be able to choose the columns shown on the issue list page, as well as their appearance order and sorting order.

Subversion

  1. FS-700 [N]: All project artifacts must be stored in a Subversion repository, and previous versions of artifacts must be saved in the repository when new versions are committed.
  2. FS-701 [N]: All users should be able to access the Subversion repository for the project.
    • The repository should be accessible by web browser, Subversion command-line and Subversion client.
    • Project members should be able to authenticate with the Subversion server by supplying their username and code.google.com password.
    • Visitors should be able to check out the project source code anonymously.
  3. FS-702 [N]: All users should be able to search the Subversion repository for the project.
    • The web browser should display a search box to search the project's trunk.

System Administrator Privileges

  1. FS-800 [M]: System administrators have the same privileges across all projects that a project administrator has on one project.
  2. FS-801 [M]: System administrators should be able to promote other users to be system administrators.
  3. FS-802 [M]: System administrators should be able to allocate additional disk space to a project.
  4. FS-803 [M]: System administrators should be able to ban a user from the service.
  5. FS-804 [N]: System administrators should be able to access statistics and metrics about a project or about the system.

Miscellaneous

  1. FS-1000 [M]: All users should be able to view another user's profile, which lists the same information displayed as when a user accesses his or her own profile.
    • A user's profile should be accessible by clicking on a user's name wherever it is listed.
